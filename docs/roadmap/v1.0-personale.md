# PaP Personale — Scope v1.0.0

> **SKU**: Prompt a Porter Personale (target single-user, local-first, no server obbligatorio).
> **Deliverable**: tag release `v1.0.0` — uscita beta del prodotto-utente.
> **Versione di partenza**: `v0.8.8` (2026-05-11).
> **Stima orizzonte**: 2-3 mesi se cert Authenticode Certum arriva entro fine mese.

## Filosofia v1.0

> Penelope ha finito la tela. Ulisse è tornato. Nel viaggio dalle Fasi 1 a 4 il prodotto ha acquisito tutte le capability che servono a una persona singola per gestire i propri prompt AI come asset di valore — versioning, varianti, regression testing, ricerca semantica, prompt componibili. Resta da rifinire l'esperienza e mettere in sicurezza la distribuzione. Quando arriva v1.0, PaP smette di essere "in beta" per l'utente individuale.

## Cosa v1.0 promette all'utente

1. **Installazione zero-friction** su Windows: NSIS per-user senza richieste UAC, auto-update silenzioso che applica patch in background, Authenticode signing che elimina SmartScreen warning su prima esecuzione.
2. **Editor curato**: tutte le scorciatoie + UI Editor "Crea variante" inline, doppia vista Sorgente/Compilato integrata (non più modale separato), markdown export con front-matter e import roundtrip.
3. **Prompt componibili evoluti**: sintassi `{{import "x" with k=v}}` (variabili scopate) + `{{import "x" version=N}}` (pinning a versione storica). Editor con hover preview e Ctrl+click navigazione già live in v0.7.0.
4. **Accessibilità di base**: 0 warning a11y in `svelte-check` (oggi ~37).
5. **Stabilità**: coverage Rust client ≥ 80%, TS client ≥ 70%, regression suite verde, security audit pulito.
6. **Documentazione utente completa**: getting started, casi d'uso comuni, troubleshooting, glossario sintassi.

## Cosa v1.0 NON promette (è in v2.0)

- Multi-utente, workspace team
- Sync server obbligatorio (resta opzionale)
- SSO / OIDC / SAML
- E2E encryption
- Web app / Browser extension
- Approval workflow + RBAC cartelle
- MCP HTTP/SSE (resta solo stdio)
- API pubblica server-side

## Strategia di branching v1.0

Sequenziale macro, parallelo micro:

- **Tutto su `main`**, niente branch lungo `feat/v1.0`.
- Una PR per ogni sub-deliverable, mergiata appena CI verde.
- Versioning durante lo sprint: continua patch line `v0.9.x` (es. `v0.9.0` "polish + auto-update", `v0.9.1` "import scopati", etc.). Il tag `v1.0.0` arriva solo quando il quality gate è ✅ chiuso.
- Bump a `v1.0.0` è un PR dedicato (`chore(release): bump v1.0.0`) con CHANGELOG narrativo.

### Parallelizzazione agenti consentita

Agenti possono lavorare in parallelo su fronti disgiunti:

| Agente | Layer | File principali |
|---|---|---|
| A — UI polish | Frontend Svelte | `apps/client/src/lib/superfici/*`, `*/components/*` |
| B — Auto-update | Packaging + signing | `.github/workflows/release.yml`, `tauri.conf.json`, NSIS template |
| C — Docs | `docs/utente/` | Mai tocca codice |
| D — Coverage | Test backend Rust + TS | `apps/client/src-tauri/src/*.rs` (test moduli), `apps/client/src/lib/*.test.ts` |

Conflitti attesi sui file condivisi (`lib.rs` invoke_handler, `ImpostazioniModal.svelte`, migration numbering) sono risolvibili con rebase incrementale dato che le PR sono piccole.

### Parallelizzazione agenti NON consentita

- Agente "v2.0 Enterprise" che scrive codice di produzione **PRIMA** di v1.0 chiusa. Resta solo design/spike (vedi §"Stream parallelo v2.0" sotto).

## Stream parallelo v2.0 — design only

Mentre v1.0 procede, un agente B può preparare strategicamente v2.0 producendo **solo artefatti non-codice di produzione**:

**Permesso**:
- Design doc in `docs/roadmap/v2.0-enterprise/*.md`
- Spike tecnici in `spikes/v2-*/` (codice esplorativo, mai mergato in `main` come feature)
- Valutazione tecnologica (es. confronto `modernc.org/sqlite` vs `mattn/go-sqlite3` per Step 0a)
- Architecture decision records in `docs/architettura/decisioni/v2-*.md`
- Mockup UI per nuove sezioni (SSO settings, E2E settings, etc.)

**Vietato**:
- Modifiche a `apps/client/`, `apps/server/`, `apps/mcp-server/`, `apps/cli/` in `main`
- Modifica a `package.json` / `Cargo.toml` per aggiungere deps "anticipate"
- Migration SQL nuove
- Nuovi cmd Tauri

Razionale: ogni byte aggiunto a `main` per v2.0 oggi è **sovra-ingegnerizzazione preventiva** finché non c'è cliente reale. Lo dichiara `fase-5-enterprise.md:19`.
