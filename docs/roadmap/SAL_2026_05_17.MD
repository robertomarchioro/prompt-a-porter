## 1. Roadmap ancora da fare

### 1.1 Patch line `v0.2.x` — Auto-update silenzioso 🔒

**Cosa**: Step 5 di Fase 2 — NSIS installer per-user (no-UAC), Tauri Updater integrato, `latest.json` firmato, downgrade refuse, signature mismatch refuse, firma Authenticode su tutti gli eseguibili.

**Bloccante**: certificato Authenticode Certum OSS — procedura KYC in corso. Senza la firma valida l'updater non può verificare la signature dei nuovi bundle, e Windows SmartScreen continua a marcare i nostri portable come "untrusted" su prima esecuzione.

**Cosa atterra appena lo sblocco cade**:
1. Branch `feat/auto-update` con NSIS per-user (target `v0.2.2`).
2. Updater + signing pipeline (target `v0.2.3`).
3. Doc operativa: `docs/auto-update.md` + `docs/decisioni/authenticode-signing.md`.

**Stima sblocco**: dipende da Certum, indicativamente 2-6 settimane post-completamento KYC.

### 1.2 Fase 5 — Ecosistema Enterprise

Vedi [`fase-5-enterprise.md`](./fase-5-enterprise.md). 9 step (0a + 1-8), tutti **domanda-driven** tranne Step 0a che è prerequisito tecnico per gli altri.

#### Step 0a — Server Go cross-OS senza Docker 🔧

**Cosa**: refactor server `papsync` per single static binary + service nativo Win/Linux/macOS. Switch driver SQLite `mattn/go-sqlite3` → `modernc.org/sqlite` (pure Go, zero CGO). Cross-compile matrix, `.deb`/`.rpm`/NSIS installer, Windows Service + systemd unit.

**Bloccante**:
- Tecnico: nessuno, lavoro fattibile oggi.
- Strategico: ha valore solo con workspace team in produzione che usa il server. Iniziarlo prematuramente espone alla **sovra-ingegnerizzazione preventiva** che è il rischio principale dichiarato di Fase 5.

**Quando partire**: appena il primo cliente/utente chiede deploy non-Docker su Win Server. Stima vuota oggi.

#### Step 1-8 — Feature enterprise 🔒/🔧

| Step | Tema | Bloccante |
|---|---|---|
| 1 | Commenti + reazioni sui prompt | Utenti team che lo chiedono |
| 2 | OIDC + SAML 2.0 SSO | Cliente enterprise paying |
| 3 | Webhook system | Step 0a (server cross-OS in produzione) |
| 4 | API pubblica + token scopati | Step 0a |
| 5 | E2E encryption workspace high-sensitivity | Cliente target (legal/healthcare/finance) reale |
| 6 | Web app (subset client) | Domanda concreta — costoso da mantenere |
| 7 | Browser extension Manifest V3 | Domanda concreta — costoso da mantenere |
| 8 | Sync Git per prompt CI/CD | Step 0a |

Tutti gli step Fase 5 sono **gated da domanda reale** (regola dichiarata in `fase-5-enterprise.md`): se non c'è utente che lo chiede, lo step viene rinviato a 1.x.

#### Tag `v1.0.0` — Uscita beta

**Bloccante**: completamento sottoinsieme minimo Fase 5 sufficiente a far uscire PaP da beta. Composizione precisa dipende da chi sarà il primo cliente serio: scenario "team interno + workspace pubblico" richiede 0a+1+3+4+8; scenario "enterprise regolato" richiede inoltre 2+5.

## 4. Note operative correnti

### Architettura

- Backbone logging: `tauri-plugin-log` (ufficiale), file rotation 5MB × KeepAll, target LogDir + Stdout + Webview con JS `attachConsole()`. Toggle ON/OFF runtime via cmd `debug_log_imposta_livello` (no riavvio). Path Linux `~/.local/share/com.pap.app/logs/pap.log`, Windows `%APPDATA%\com.pap.app\logs\pap.log`, macOS `~/Library/Logs/com.pap.app/`.
- Sezione **Impostazioni → Sviluppo** estensibile: pattern card con toggle + actions inline. Prima feature live: Debug log. Slot futuri: AI suggestions beta, vector debug, etc.
- Sintassi segnaposti: `{{nome}}` normale + `{{globale nome}}` (v0.8.5) con resolver separato + pre-fill DB + auto-UPSERT.

### Workflow di rilascio

- Pattern multi-PR (sub-PR feature + bump finale + tag → release.yml) confermato come modus operandi standard per sprint ≤ 5 issue.
- Modalità autonoma autorizzabile su richiesta esplicita utente (es. F8-F11 redesign, v0.8.5, v0.8.7) — auto-merge su CI verde, stop solo su CI rossa o dubbio architetturale.
- `tauri-action` crea release in **Draft** di default. Promozione a Latest published è manuale (verifiche smoke + decisione di shipping).

### Sicurezza

- Security audit settimanale (`security-audit.yml`, cron lunedì 06:00 UTC) verde dopo bump Go 1.25.10 (PR #164) + chi v5.2.5 + golang.org/x/crypto v0.51.0 (PR #166). Issue tracking #165 chiusa.
- Workflow security-audit usa `go-version: "1.25.10"` pinnato esplicito per evitare regressioni manifest cache.
