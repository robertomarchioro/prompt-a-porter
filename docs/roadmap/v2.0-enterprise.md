## Filosofia v2.0

> Quando Ulisse torna a Itaca, riconosce la sua casa ma sa anche che il mondo intorno è cambiato. Non basta più la sala del trono: serve una rete di alleanze.

In v2.0 PaP smette di essere "la mia libreria di prompt" e inizia a essere "il nodo della mia infrastruttura AI". Le decisioni qui hanno **impatto architetturale** sui dati esistenti — la cifratura E2E in particolare richiede attenzione perché modifica il modo in cui server e client si scambiano informazioni.

**Il rischio principale di v2.0 è la sovra-ingegnerizzazione preventiva.** Ogni step deve avere risposta affermativa alle 3 domande di Fase 5:
1. C'è un utente o cliente reale che lo chiede?
2. Il costo di mantenimento è giustificato dal volume d'uso atteso?
3. Esiste un'alternativa standard più semplice che copre l'80% del caso?

Se le risposte sono "no", lo step viene rinviato a 2.x o cassato.

## Cosa v2.0 promette al cliente Enterprise

Subset minimo che farebbe uscire la linea Enterprise da beta — sufficiente a chiudere il primo contratto realistico:

1. **Server cross-platform single binary** (Linux/Win/macOS), deployabile come servizio nativo senza Docker. È il prerequisito di tutto il resto.
2. **SSO via OIDC/SAML 2.0** per autenticazione enterprise.
3. **Approval workflow + RBAC per cartelle** — controllo di chi può approvare prompt, chi può modificarli.
4. **API pubblica documentata** (OpenAPI 3.1) + token scopati per integrazione CI/CD.
5. **Sync Git per prompt** — versionare prompt come codice in repo Git aziendale.
6. **Webhook system** — notifiche eventi (nuovo prompt, rating, approval) verso sistemi esterni.

## Cosa v2.0 promette solo se richiesto

Feature ad alto costo di mantenimento, gated da cliente target esplicito:

7. **E2E encryption** per workspace high-sensitivity (legal/healthcare/finance/defense).
8. **Web app** (subset funzionalità client).
9. **Browser extension** Manifest V3 + Native Messaging.

## Strategia SKU multi-SKU post-v2.0

Dopo l'apertura della linea Enterprise, PaP avrà due binari distribuiti:

| Binario | Target | Tier | Distribuzione |
|---|---|---|---|
| `pap` | Single user | Personale (free) | GitHub release Latest |
| `pap-enterprise` | Workspace team | Enterprise (commerciale) | Canale dedicato (TBD: licensing portal, marketplace, etc.) |

### Architettura single-codebase

Stessa codebase con **feature flag Cargo + dynamic import Svelte**:

- Backend Rust: `#[cfg(feature = "enterprise")] pub mod sso;` etc. Build con `cargo build --features enterprise` produce `pap-enterprise`.
- Frontend Svelte: componenti enterprise dietro flag runtime `__PAP_ENTERPRISE__` (settato a build-time via Vite env). UI nasconde sezioni che non si applicano.
- Schema DB: tabelle enterprise (Comments, Reactions, FolderPermissions, AuditLog esteso) hanno migration condizionali — esistono in entrambi i binari ma restano vuote in Personale.
- Tauri capabilities: capability set distinto per Enterprise (più permessi: filesystem, http, etc.).

### Rilasci di pari passo

Dopo v2.0.0:

- Patch line `v1.x.x` per PaP Personale continua su `main`.
- Patch line `v2.x.x` per PaP Enterprise può vivere su `main` (single codebase) — i tag distinguono.
- CI build matrix: 2 artifact per OS (personal/enterprise), 6 totali (Win/Linux/macOS × 2).
- Esempio sprint condiviso: feature `recovery-on-folder-rename` atterra in `main`, taggata sia `v1.1.0` (binario personal) che `v2.1.0` (binario enterprise) lo stesso giorno.

### Cosa cambia per i contributor

- Una PR può toccare codice "comune" (sia in personal che enterprise) o "enterprise-only".
- File pattern `*_enterprise.rs` per moduli esclusivi enterprise.
- Reviewer abilitati a flaggare `enterprise-feature` per applicare check addizionali (E2E, security, license).
